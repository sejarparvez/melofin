# Detail Views & Hover Play Icons Plan

## Overview

Currently, clicking any card (song, playlist, album) immediately plays it. This plan adds:
1. **Click вҶ’ details page** for ALL cards (songs, playlists, albums)
2. **Hover вҶ’ play icon overlay** on cards for quick play without navigating away
3. **Full metadata detail page** with track list, description, year, "Play All" button

---

## Part 1: MediaKind Detection

**Problem:** The `Track` struct has no way to distinguish songs from playlists/albums. The URL pattern is the only signal: `watch?v=` = song, `browse/` = playlist/album.

**Approach:** Add a `MediaKind` enum and helper in `src/search.rs`. No changes to the Track struct itself.

```rust
#[derive(Clone, Debug, PartialEq, Eq)]
pub enum MediaKind {
    Song,
    Playlist,
    Album,
    Artist,
}

impl Track {
    pub fn media_kind(&self) -> MediaKind {
        if self.url.contains("/browse/") {
            // Could be playlist, album, or artist вҖ” default to Playlist
            // for now; detail view will show the actual type from metadata
            MediaKind::Playlist
        } else {
            MediaKind::Song
        }
    }

    pub fn browse_id(&self) -> Option<&str> {
        self.url.strip_prefix("https://music.youtube.com/browse/")
    }
}
```

- `src/search.rs` вҖ” add `MediaKind` enum, `media_kind()`, `browse_id()` methods

## Part 2: Detail Data Fetcher

**New file: `src/detail_fetch.rs`**

Fetches playlist/album/artist details from InnerTube and returns a structured result.

### Data structures

```rust
pub struct DetailMetadata {
    pub title: String,
    pub artist: String,
    pub thumbnail_url: String,
    pub description: String,
    pub year: String,
    pub track_count: usize,
}

pub struct DetailResult {
    pub metadata: DetailMetadata,
    pub tracks: Vec<Track>,
}
```

### Fetching logic

Uses the existing `browse_request()` with the browse_id, then parses:

1. **Header metadata** from `musicResponsiveHeaderRenderer` (or `musicImmersiveHeaderRenderer`):
   - Title: `.../title/runs[0].text`
   - Artist: `.../subtitle/runs[0].text` or `.../description/runs[0].text`
   - Thumbnail: `.../image/musicThumbnailRenderer/thumbnail/thumbnails[-1].url`
   - Description: `.../description/bodyBodyRenderer/runs[].text` (join runs)
   - Year: extracted from subtitle runs or description

2. **Track list** from `musicShelfRenderer`:
   - Reuse the existing `parse_shelf_items()` pattern from `liked_songs.rs`
   - Extract `musicResponsiveListItemRenderer` items with `videoId`, title, artist, duration

3. **Continuation** for large playlists:
   - Follow `musicShelfRenderer/continuations/0/nextContinuationData/continuation`
   - Same pattern as `liked_songs.rs`

### Functions

```rust
pub fn fetch_detail(
    cookies_path: &Path,
    browse_id: &str,
) -> Result<DetailResult>
```

**Files to create:**
- `src/detail_fetch.rs`

**Files to change:**
- `src/lib.rs` вҖ” add `pub mod detail_fetch;`
- `src/innertube.rs` вҖ” make `parse_shelf_items()` `pub(crate)` so `detail_fetch.rs` can reuse it

## Part 4: Navigation Stack

**Problem:** The `content_stack` currently only has "home" and "liked". Need to push/pop detail views.

**Approach:** Maintain a `Vec<String>` of active detail view names. Push adds a new named child, pop removes it.

### In `window.rs`

```rust
let detail_stack: Rc<RefCell<Vec<String>>> = Rc::new(RefCell::new(Vec::new()));
let mut detail_counter: usize = 0;

// Push function
fn push_detail(stack: &gtk::Stack, name: &str, child: &impl IsA<gtk::Widget>) {
    stack.add_named(child, Some(name));
    stack.set_visible_child(child);
}

// Pop function
fn pop_detail(stack: &gtk::Stack, name: &str) {
    if let Some(child) = stack.child_by_name(name) {
        stack.remove(&child);
    }
    // Show previous child
}
```

The "home" and "liked" views always stay in the stack. Detail views are pushed on top and removed on back.

### Callback wiring

The `on_select` callback in each view changes from `Fn(Track)` to:

```rust
// For songs: navigate to detail page (which shows the song's context)
// For playlists/albums: navigate to detail page
on_select: Box<dyn Fn(Track)>,
on_play: Box<dyn Fn(Track)>,  // direct play (for hover icon)
on_back: Box<dyn Fn()>,
```

## Part 5: Hover Play Icon

**Problem:** GTK4 doesn't have CSS `:hover` that can toggle widget visibility. Need a programmatic approach.

**Approach:** Wrap card art in a `gtk::Overlay` with a hidden play button. `EventControllerMotion` toggles visibility on enter/leave.

### Implementation in `home_view.rs` `track_card()`

```rust
// Wrap art_frame in an Overlay
let overlay = gtk::Overlay::new();
overlay.set_child(Some(&art_frame));

let play_button = gtk::Button::from_icon_name("media-playback-start-symbolic");
play_button.add_css_class("circular");
play_button.add_css_class("suggested-action");
play_button.set_halign(gtk::Align::End);
play_button.set_valign(gtk::Align::End);
play_button.set_margin_end(8);
play_button.set_margin_bottom(8);
play_button.set_visible(false);
overlay.add_overlay(&play_button);

// Hover handling
let motion = gtk::EventControllerMotion::new();
let play_button_ref = play_button.clone();
motion.connect_enter(move |_, _, _| {
    play_button_ref.set_visible(true);
});
let play_button_ref = play_button.clone();
motion.connect_leave(move |_| {
    play_button_ref.set_visible(false);
});
card.add_controller(motion);

// Play button clicks play directly (not navigate)
play_button.connect_clicked(move |_| {
    on_play(track.clone());
});
```

The card itself still navigates to detail on click. The play button overlay intercepts its own click and plays directly.

### CSS styling

Add to `style.css`:
```css
.play-overlay button {
    opacity: 0;
    transition: opacity 200ms;
}
.play-overlay:hover button {
    opacity: 1;
}
```

Actually, since we're using `EventControllerMotion` for visibility, the CSS just needs to style the button's position (absolute bottom-right within the overlay).

## Part 6: Callback Signature Changes

The current `on_select: impl Fn(Track) + 'static + Clone` pattern needs to become richer. Since closures with multiple callbacks are awkward, introduce a helper struct:

```rust
// In window.rs or a new ui/navigation.rs
pub struct Navigation {
    pub content_stack: gtk::Stack,
    pub play_track: Box<dyn Fn(Track)>,
    pub detail_counter: Rc<Cell<usize>>,
    pub detail_stack: Rc<RefCell<Vec<String>>>,
}

impl Navigation {
    pub fn navigate_to(&self, track: &Track) { ... }
    pub fn play(&self, track: &Track) { ... }
    pub fn go_back(&self) { ... }
}
```

This keeps the callback plumbing clean across all views.

- `src/ui/navigation.rs` вҖ” or inline in `window.rs`
